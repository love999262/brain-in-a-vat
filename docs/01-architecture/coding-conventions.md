# 编码约定

## 目的

本文档用于统一工程风格，减少多人协作或多次 AI 会话带来的实现漂移。

## 通用原则

- 业务边界优先于语法偏好
- 能通过类型表达的约束，优先通过类型表达
- 能通过文档和 schema 固定的内容，不要在代码里重复发明一套
- 优先写清晰、可读、可追踪的代码，而不是省几行代码

## 命名约定

### 包名

- 包目录统一使用小写短横线命名
- 例如：`engine-core`、`browser-runtime`

### 文件名

- 文档文件使用小写短横线命名
- 代码文件遵循所在技术栈惯例
- 类型文件、协议文件、schema 文件命名应具备明确含义，不使用 `misc`、`temp`、`helper2` 之类名称

### 标识符

- TypeScript 类型、类、接口使用大驼峰
- 变量、函数使用小驼峰
- 常量使用大写下划线或语义化导出常量
- 事件名、命令名、状态码统一使用稳定字符串，不拼接随意缩写

## 目录约定

- `apps` 只放壳层应用
- `packages` 只放可复用模块
- `worlds / characters / sources` 只放内容资产
- `docs/examples` 只放示例，不放真实运行内容

## 导出约定

- 每个包应有明确的公开导出面
- 不要把内部实现文件默认全部导出
- UI 层不得导入引擎内部私有模块
- 宿主层只依赖公开 API

## 类型约定

- Command、Event、ViewModel、Pack 配置都必须有明确类型
- 不使用 `any` 逃避边界问题
- 对外协议优先使用稳定字段，不依赖隐式约定

## 测试约定

- schema、协议、引擎最小闭环优先测试
- 新增公开协议时，应同步补充最小示例或测试
- 修复行为缺陷时，应优先补回归测试

## 文档约定

- 任何改动公共协议、目录结构、V1 范围、Pack 结构之前，先改文档
- 任何完成后的代码改动，都必须做一次“文档与实现一致性审计”
- 每次完成一部分工作后，必须更新 `docs/09-roadmap/execution-status.md`
- 示例文件变动后，应保证与当前文档描述一致

### 文档与实现一致性审计

每次代码修改完成后，至少检查以下内容：

- `README.md` 中的“当前实现”“快速开始”“仓库结构”是否仍然准确
- `docs/00-overview/overview.md` 中的架构边界、目录结构、能力说明是否仍然准确
- `docs/09-roadmap/execution-status.md` 中的阶段、当前主线、已完成事项、下一步任务是否仍然准确
- 受影响的专题文档是否仍然准确
- `docs/examples` 中的示例是否仍然和当前协议、Pack 结构一致

如果检查后发现不一致，先改文档，再提交代码。

## 明确禁止

- 在 UI 中持有业务真相
- 在 Provider 中写角色或世界规则
- 在 Renderer 中推导业务关系
- 未更新文档就直接改公开边界
