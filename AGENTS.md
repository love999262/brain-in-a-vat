# AGENTS.md

本文件用于约束任何进入本仓库协作的 AI 助手。除非用户明确要求跳过文档流程，否则所有开发工作都必须先遵守这里的阅读顺序和更新规则。

## 1. 强制阅读顺序

开始任何分析、设计、编码、重构、补文档之前，必须先阅读以下文件：

1. `/README.md`
2. `/docs/00-overview/overview.md`
3. `/docs/09-roadmap/execution-status.md`

如果本次任务会修改某个已存在的专题文档，还必须继续阅读对应专题文档。

如果专题文档还不存在，但本次工作已经明显进入该主题范围，应先补文档骨架，再开始代码开发。

如果本次工作已经进入具体实现阶段，还必须继续阅读：

- `/docs/01-architecture/coding-conventions.md`
- `/docs/09-roadmap/phase-0-checklist.md` 或 `/docs/09-roadmap/phase-1-checklist.md`
- `/docs/examples/README.md`

## 2. 项目定位

本项目是一个前端优先、本地优先、可插拔、可嵌入的多角色演化引擎。

请始终牢记：

- 引擎是项目本体，UI 不是
- 内容包是配置资产，不是硬编码逻辑
- `serverless` 只能增强，不能成为默认前提
- V1 只有一个官方壳层：`Standalone App`
- V1 只公开一个 Web Component：`<brain-vat-world>`

## 3. 开发前检查

开始工作前，必须先确认以下事项：

- 当前任务是否符合 `/docs/09-roadmap/execution-status.md` 中的当前阶段
- 任务是否已经存在、是否已经有人在做、是否已完成
- 本次工作会影响哪些文档、哪些包、哪些公共协议
- 本次修改是否会破坏总纲里已经冻结的 V1 决策

如果任务与总纲冲突，不要直接实现，应先更新文档并说明变更理由。

如果任务与示例文件冲突，也必须同步更新示例文件。

## 4. 开发流程

所有开发工作必须按这个顺序执行：

1. 阅读必读文档
2. 确认本次任务对应的阶段与条目
3. 如有必要，先更新设计文档
4. 再修改代码
5. 完成后更新执行状态文档

禁止跳过第 5 步。

## 5. 执行状态文档规则

`/docs/09-roadmap/execution-status.md` 是当前仓库唯一的进度真相。

每次完成一部分工作后，必须同步更新：

- 当前阶段
- 当前进行中的任务
- 已完成事项
- 下一步事项
- 变更记录

如果修改了架构、协议、目录结构或 V1 范围，也必须同步更新相关设计文档，而不只更新状态文档。

## 6. 状态标记规范

执行状态文档中的任务统一使用以下状态：

- `TODO`
- `IN_PROGRESS`
- `DONE`
- `BLOCKED`

同一时间只能有少量任务处于 `IN_PROGRESS`，不要把整页任务同时标成进行中。

## 7. 文档优先级

当多个文件出现不一致时，优先级从高到低如下：

1. `/docs/00-overview/overview.md`
2. `/docs/09-roadmap/execution-status.md`
3. 相关专题文档
4. `/README.md`

如果低优先级文档与高优先级文档冲突，应以高优先级文档为准并补齐同步更新。

## 8. 编码约束

- 不要把角色、消息源、世界写死在代码里
- 不要让 UI 直接持有业务真相
- 不要让 provider 承担业务决策
- 不要让 renderer 理解业务关系和记忆
- 不要在 V1 中擅自扩展出第二个官方壳层或第二个公开组件

## 9. 文档缺失时的处理

如果本次工作必需的设计文档缺失：

- 先补最小文档骨架
- 再开始实现

不要在没有设计落点的情况下直接编码。

## 10. 完成定义

一次任务只有在以下条件同时满足时才算真正完成：

- 代码或文档本身已经完成
- 相关测试或验证已经执行，或明确记录未执行原因
- `/docs/09-roadmap/execution-status.md` 已更新
- 如有必要，相关专题设计文档已同步更新

如果没有更新执行状态文档，这次任务视为未完成。
